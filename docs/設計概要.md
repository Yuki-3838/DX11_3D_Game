# DX11 3D Game 設計概要

このプロジェクトは、敵の行動を読んで、攻撃後のすきを突く一対一アクションの土台です。
最初から100体を作るのではなく、100体を安全に増やせるデータ、計測、デバッグの流れを先に固定します。

## 現在の実行フロー

1. `Main.cpp` が `GameApp` を作る。
2. `GameApp::Init` がWin32ウィンドウ、DX11 Device、SwapChain、RenderTargetViewを初期化する。
3. `GameApp::Run` がWindowsメッセージを処理しながら毎フレーム `ProcessFrame` を呼ぶ。
4. `FrameTimer` が60FPS固定更新のための蓄積時間を管理する。
5. `GameApp::FixedUpdate` が攻撃フェーズを進める。今はサンプル攻撃の前兆、攻撃判定、硬直、クールダウンを循環させる。
6. `GameApp::Draw` がバックバッファをクリアする。今後メッシュ描画や影は `Renderer` へ移す。
7. `PerformanceProfiler` がフレーム時間、FPS、固定更新回数を記録する。
8. `DebugOverlay` がImGuiへ渡す表示モデルを保持する。
9. `BrowserDebugReporter` が `debug/debug_state.json` と `debug/index.html` を出力する。

## 戦闘データの考え方

`CombatDesign.h` は、攻撃を以下の区間で扱います。

- `Anticipation`: 前兆。プレイヤーが読んで回避やガードを選ぶ時間。
- `Active`: 攻撃判定が出る時間。
- `Recovery`: 反撃のすき。
- `Cooldown`: 次行動までの制限時間。
- `Commit`: 一度始めると中断できない時間。

命中判定は2段階です。

- `BroadPhaseFilter`: 距離、角度、状態で候補を絞る。
- `HitboxDefinition` と `NarrowPhaseType`: 球、カプセル、OBBなどで精密判定する。

この構造にしておくと、敵ごとの攻撃やAIの違いをコード分岐ではなくデータ差分で増やせます。

## デバッグ表示

Dear ImGuiは `ImGuiDebugAdapter` に閉じ込めています。
`DX11_GAME_ENABLE_IMGUI` を定義し、Dear ImGui本体とDX11/Win32バックエンドを追加すると、
`DebugOverlay` の値をゲーム画面内に表示できます。

ブラウザ確認はゲーム起動中に生成される `debug/index.html` を開きます。
同じフォルダに更新される `debug/debug_state.json` を読み、FPS、フレーム時間、攻撃フェーズ、命中候補数を表示します。
